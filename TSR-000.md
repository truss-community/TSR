## Abstract

This TSR defines the TSR (Truss Standard Request) process itself — a formal mechanism for proposing, discussing, and adopting standardized changes to the Truss programming language, its standard library, tooling, or governance.

## 1. What is a TSR?

A TSR (Truss Standard Request) is a **design document** that describes a potential change to Truss. It is the primary vehicle for:

- Language syntax or semantics changes
- Standard library additions or modifications
- Toolchain specifications (formatter, LSP, build system)
- Process or governance changes

## 2. TSR Roles

| Role | Responsibility |
|------|----------------|
| **Author** | Writes and shepherds the TSR |
| **Reviewer** | Core team member ensuring technical and procedural quality |
| **Final Decision Maker** | Language Lead or Technical Committee |
| **Community** | Provides feedback, discussion, and non-binding consensus |

## 3. TSR Lifecycle

**Draft** -> **Proposal** -> **Discussion** -> **Review** -> **Final Decision** -> (if accepted) **Implementation** -> **Active**

| Stage | Description |
|-------|-------------|
| **Draft** | Author writes the TSR following the format in §4 |
| **Proposal** | Submitted to official TSR repository; an ID is assigned |
| **Discussion** | Public discussion for at least 14 days |
| **Review** | Reviewers request changes or raise concerns |
| **Final Decision** | Accepted, Rejected, or Deferred by decision maker |
| **Implementation** | Merged into language/library/toolchain (if accepted) |
| **Active** | Released as part of an official Truss version |

## 4. TSR Document Format

Each TSR is a **Markdown file** with the following required sections (`*` = required):

```markdown
# TSR-NNN: Title

- **Author**: Name <contact>
- **Type**: Standards Track / Informational / Process
- **Status**: Draft | Proposal | Discussion | Review | Accepted | Rejected | Deferred | Active
- **Created**: YYYY-MM-DD
- **Updated**: YYYY-MM-DD (optional)
- **Requires Truss**: e.g., 2.0+

## Abstract
* (1–3 sentences summarizing the proposal)

## Motivation
* (Why is this change needed? What problems does it solve?)

## Detailed Design
* (Syntax, semantics, APIs, examples)

## Rationale
* (Why this design over alternatives? Trade-offs?)

## Backward Compatibility
* (Breaking changes? Migration plan?)

## Reference Implementation
(Optional but recommended)

## Open Questions
(Points requiring further discussion)

## References
(Links to related PRs, discussions, previous TSRs)
```

## 5. TSR Types

| Type | Purpose |
|------|---------|
| **Standards Track** | Language or standard library changes |
| **Informational** | Design principles, best practices, non-mandatory guidance |
| **Process** | Changes to the TSR process or governance rules |

## 6. TSR Status Definitions

| Status | Meaning |
|--------|---------|
| **Draft** | Author is actively writing; not yet submitted |
| **Proposal** | Submitted, waiting for review and ID assignment |
| **Discussion** | Open for community feedback |
| **Review** | Under active review by core team |
| **Accepted** | Approved; awaiting implementation |
| **Rejected** | Declined (reason documented) |
| **Deferred** | Postponed to future consideration |
| **Active** | Implemented and released |

## 7. Numbering and Naming

- Format: `TSR-NNN` where `NNN` is a monotonically increasing integer.
- Numbers are assigned by a reviewer **once** the TSR enters the Proposal stage.
- Rejected or Deferred TSRs **retain their numbers** (no reuse).

## 8. Decision Criteria

A TSR is accepted when it meets all of the following:

1. **Clear and Understandable** — No ambiguity for community or core team
2. **Technically Feasible** — A reasonable implementation path exists
3. **Consistent with Truss Philosophy**:
   - Explicit over implicit
   - Practical over pure
   - Simplicity is not sacrificed lightly
4. **Ecosystem Benefit** — The problem has broad or significant value

## 9. TSR vs. GitHub Issue

| Aspect | TSR | Regular Issue |
|--------|-----|----------------|
| **Scope** | Language-level or major ecosystem changes | Small features, bug fixes |
| **Process** | Formal (Draft → … → Active) | Fast, lightweight |
| **Documentation** | Structured with required sections | Free form |
| **Discussion Period** | Minimum 14 days | As needed |
| **Decision Body** | Language Lead / Tech Committee | Maintainer discretion |
